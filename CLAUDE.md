## Project Overview
InternTrack is a capstone OJT/internship monitoring system for Mater Dei College (Bohol, Philippines), built by a 4-person team (Group 1). It digitizes SIPP internship processes: student profiles, attendance/clock-in, weekly activity journals, evaluations, and reporting.

This is a **monorepo** with three parts:
- **repo root** â€” Laravel 13 REST API (PHP 8.3, MySQL/SQLite). There is no `api/` subfolder â€” the Laravel app lives at the top level (`app/`, `routes/`, `database/`, etc.)
- `web/` â€” Vue 3 SPA (Vite, Tailwind CSS v4, **TypeScript** â€” `.ts` files, `tsconfig.json`, and the `typescript` package are already in use under `web/src`)
- `mobile/` â€” React Native / Expo app (Expo SDK 56, TypeScript, expo-router). Fixed: the button is now labeled "Deactivate", and an inactive row shows "Reactivate" instead (calling the pre-existing `activate()` endpoint) â€” no backend changes needed, both routes already existed. `AdminUsersPage.vue`'s role filter now offers **All Users** (default) / Student / Coordinator / Supervisor (Supervisor was missing entirely before), and `Admin/UserController::index()` unconditionally excludes `role = 'admin'` regardless of filter â€” this page manages day-to-day accounts, not the single admin account itself (`test_index_never_returns_admin_accounts` in `UserControllerTest.php`). Separately confirmed (no code change needed): a completed student's write access is **already** blocked today â€” `ResolvesStudentEnrollment::activeEnrollment()` (used by every student write endpoint) requires `status = 'active'`, which a `completed` enrollment row never matches, while `currentEnrollment()` (used by reads) allows `active` **or** `completed` â€” so "restrict a graduated student to read-only" was already the system's behavior via enrollment status, not `users.is_active`.
- Creating a `student`-role user auto-creates a matching `student_profiles` row via `UserObserver` (on the `User` model's `created` event), with a placeholder `student_id_number` (`PENDING-XXXXXXXX`) until the real registrar ID is filled in.

## Roles & Configuration
- Admin creates accounts (`Admin/UserController`) and global structure (departments, programs, companies). Creating a student user does **not** enroll them â€” see below.
- Coordinators own `batches` for their program(s) (`Coordinator/BatchController`) and enroll students into a batch (`Coordinator/EnrollmentController`), which creates the `batch_students` row. `batch_students` is the **authoritative studentâ†”company/supervisor linkage**, with a **DB-level `UNIQUE(batch_id, student_id)`** backstop â€” at most one row per (batch, student) pair, ever; a row's `status` (`active`/`completed`/`dropped`) carries its lifecycle. Student-facing **WRITE** endpoints (journal/weekly-log save + submit, info sheet, weekly activity logs) require an active (`status='active'`) row via `ResolvesStudentEnrollment::activeEnrollment()` and 422 with "You are not currently enrolled in an active OJT batch." without one; journal/weekly-log **READ** endpoints (show/index/calendar/PDF) instead use `currentEnrollment()` (latest `active` OR `completed`), so a completed student keeps full read access. **All enrollment placement flows through `App\Services\EnrollmentService::enrollOrReactivate()`** (used by `EnrollmentController::store`, info-sheet Accept, and `BatchRosterController::add`): it reuses ANY existing row for the exact (batch, student) pair regardless of status â€” refreshing `company_id`/`supervisor_id`/`assigned_division` and setting it back to `active` â€” and only creates a fresh row when none exists, which is what makes the unique index safe.
- **The OJT journal window is REAL-TIME, not estimate-bound** (`ResolvesStudentEnrollment::ojtRange()`): start = the batch's `start_date`, end = `batch_students.completed_at` once the coordinator marks the enrollment completed, or **today** (rolling) while it's still active. The info sheet's `ojt_info.ojt_start_date`/`ojt_end_date` are **informational only** â€” they render on the form/PDF but gate nothing. The **completion signal** is coordinator-driven from the batch roster modal (`BatchRosterController::complete`/`reopen`, `PATCH batches/{batch}/roster/{batchStudent}/complete|reopen`, confirm-first + toast like drop/reactivate): `complete` requires an active row (422 otherwise) and sets `status='completed'`; `reopen` requires a completed row and is blocked if the student is active elsewhere. `completed_at` is maintained **only** by a `BatchStudent::booted()` saving hook (stamped on entry to `completed`, cleared on exit â€” `EnrollmentController::update` and every other instance save get it for free; don't set it manually). Downstream: a completed student's writes 422 (`isEditableDate` also requires `status='active'`), the journal calendar freezes its upper bound at `completed_at` (days past it are `no_entry`; while active, days after today stay `future`), the student weekly-logs list ends at the completion week (it **starts** at the Monday of the student's first daily entry or earliest weekly log in the batch â€” empty list until some work exists), `WeeklyBundlingService` skips them (it only iterates `status='active'`; their existing logs persist), and `User::isInfoSheetGated()` treats an active **or completed** enrollment as having cleared intake so completing a legacy student doesn't re-gate them.
- Coordinators are assigned to a department via the `coordinator_departments` pivot (`User::departmentsCoordinated()` / `Department::coordinators()`), and admins manage the assignment from the Admin Departments page (`Admin/DepartmentController::assignCoordinator`/`removeCoordinator`). **As of the 2026-07-12 admin migration (`enforce_single_department_per_coordinator`), this is one department per coordinator** â€” `coordinator_departments.coordinator_id` is now unique, `AssignCoordinatorRequest` rejects assigning a coordinator already attached elsewhere, and `Admin/UserController::store` takes a single `department_id` (not `department_ids[]`). This superseded the earlier "a coordinator can run multiple departments" design â€” a department can still have multiple programs and multiple coordinators, just not the reverse. A coordinator's programs are resolved via `User::coordinatorProgramIds()` â€” every program in their assigned department, merged with the programs of batches they already coordinate (`batchesCoordinated()`, kept only as a backward-safety net) â€” not `batchesCoordinated()` alone. `users.program_id` is retained (still used by account creation/`StoreUserRequest`) but is **no longer** the coordinator scoping source.
- **Enrollment is student-driven, coordinator-gated intake â€” the Student Information Sheet is the gateway** (this superseded the old "enroll first, fill sheet after" model). End-to-end flow:
  1. **Coordinator creates the student login** on the Users â†’ Interns page (`EnrollmentController::createAccount`, `CreateAccountRequest`): NAME, **USERNAME** (no email), PASSWORD, optional STUDENT ID, **PROGRAM** (scoped to their department) + **BATCH** (one they coordinate, belonging to that program). The account is created **NOT-ENROLLED**; the coordinator's intended placement is recorded as a **DRAFT `student_information_sheets` row whose `batch_id` is the intended batch** â€” the single home for "intended batch before Accept" (no new column). `enrollment-options` now also returns the coordinator's `batches` (with `program_id`) so the form filters the Batch dropdown by the chosen program. Account creation is still distinct from enrollment.
  2. **The student is GATED** until their sheet is `approved`: `EnsureInfoSheetApproved` middleware (alias `infosheet.approved`) wraps a gated student route subgroup in `routes/api.php` â€” only `student/info-sheet` (show/store), `student/info-sheet/pdf`, `student/companies`, and `student/password` stay ungated; every other student endpoint 403s. `User::isInfoSheetGated()` is the single source of truth (gated unless an approved sheet OR an active enrollment exists â€” the latter keeps legacy directly-enrolled students working), surfaced on `/api/user` as `student_gated` for the frontend router guard (`web/src/router/index.ts`) + `StudentLayout` nav filtering.
  3. **The student fills + submits the sheet** (`StudentInfoSheetPage.vue`, matching the client reference: Student Trainee Information + Internship Company Information; the on-screen form itself has no sketch input, though the exported PDF gained a blank sketch box â€” see item 5 below). `Student/StudentInfoSheetController::store()` **no longer requires an active enrollment** â€” it writes the scaffolded draft (or falls back to an active enrollment's batch for legacy students), re-derives the read-only Program/Coordinator from the intended batch (anti-spoof), and clears any prior rejection. **Name of Company is the one constrained field** â€” a dropdown from `GET student/companies` (active companies); it stores both `ojt_info.company_id` (drives Accept) and `host_company` (name). **Year Level is a constrained dropdown** (`1st Year`..`4th Year`, validated via `in:` in `StoreInfoSheetRequest`), not free text. **An approved (enrolled) sheet is no longer permanently read-only** (this supersedes the earlier "approved = fully locked" design): `store()` still blocks a fresh gate submission from replacing it, but now allows **partial post-enrollment editing** â€” **Program & Year and the assigned Company stay locked** (re-derived server-side from the existing sheet, ignoring whatever the client sent, since they drove the coordinator's Accept step), while personal info, company address, signatory, supervisor, schedule, and dates remain freely editable for routine profile upkeep. Saving an already-`approved` sheet **never** moves it back to `draft`/`submitted`/clears its `rejection_reason` â€” `submission_status` stays pinned to `approved` â€” so routine edits can't accidentally re-trip `EnsureInfoSheetApproved` and re-gate an already-enrolled student. For a not-yet-approved sheet, status flow is unchanged: `draft`/`submitted`/`rejected` are fully editable (`submission_status` enum has `'rejected'` + a nullable `rejection_reason`). Submit confirms first + toasts.
  4. **The coordinator reviews on Student Info Sheets** (`Coordinator/CoordinatorInfoSheetController`, `CoordinatorInfoSheetsPage.vue`) â€” now an **Accept/Reject queue scoped by the sheet's intended-batch program** (so NOT-yet-enrolled students finally appear; the old enrollment-only scoping couldn't surface them), with status/program filters. **Accept = enroll** (`accept`): creates the active `batch_students` row for the sheet's batch + the student's chosen company + **that company's Company Supervisor**, then marks the sheet `approved` (lifting the gate). Only a `submitted` sheet is acceptable (re-accept is a no-op 422, never double-enrolls); a chosen company with no supervisor 422s. **Reject** (`reject`, `RejectInfoSheetRequest`) sets `rejected` + a required reason shown to the student, who edits + resubmits. Both are scope-enforced (403 out of scope). Enrollment is shared via `App\Services\EnrollmentService::enrollOrReactivate()`, reused by both `EnrollmentController::store` and Accept.
  5. **Individual Info Sheet PDF** (`pdf.info-sheet` blade via the shared `BuildsInfoSheetPdf` trait): official MDC document with the logo embedded as a base64 data URI from `public/images/mdc-logo.png` (labeled-placeholder fallback if absent), the college/program header, the two labeled sections, plus a **"Sketch of Internship Company Location" box** (bordered, blank, 35mm tall) at the bottom of the page â€” reinstated in commit `b401733`, reversing the earlier "no sketch box" decision recorded in this file; the on-screen `StudentInfoSheetPage.vue` form itself still has no sketch input, this is PDF-only. Downloadable by the student (`GET student/info-sheet/pdf`, own) and the coordinator (`GET coordinator/info-sheets/{student}/pdf`, in-scope; 403 otherwise) â€” "Download PDF" buttons on both surfaces.

- **Coordinator UX rule**: crucial/destructive actions (delete a report row, deactivate a template, deactivate a batch/company, remove a supervisor/coordinator) prompt a **confirm dialog first**; successful saves show a **toast after**. Use the shared helpers in `web/src/lib/toast.ts` (`confirmAction` â€” now an **async** promise-based styled modal, NOT `window.confirm` anymore; `showToast(message, type, timeout?)`) with `web/src/components/ToastHost.vue` mounted on the page (per-page, not layout-level â€” a page using either helper must mount it itself). `ToastHost` renders a fixed top-right stack: white card, colored left border + matching icon (green check / red exclamation) per `type`, shadow, and a `TransitionGroup` fade/slide. This confirm-first-destructive + toast-after-success pattern is enforced consistently across coordinator, student, supervisor, and admin destructive actions (delete/remove/deactivate/drop) â€” not just coordinator. **Toast durations auto-scale, aren't dismissible-by-default, and are capped** (this pass hardened the previously bare-bones toast queue): `timeout` is now optional â€” when omitted, `showToast` derives it via `defaultTimeoutFor(message, type)` (3s base, +50ms per character past a 20-char grace window, capped at 8s; `error` toasts get a 4.5s floor even when short, since severity is its own importance signal, not just length) â€” none of the ~40 existing call sites pass an explicit timeout, so all of them picked up auto-scaling for free. A `MAX_VISIBLE_TOASTS = 4` cap evicts the oldest toast (clearing its timer) once exceeded, so a burst of actions can't stack the queue indefinitely. Each toast now has a close (Ã—) button (`dismissToast(id)`) and pauses its countdown on `@mouseenter`/resumes on `@mouseleave` (`pauseToast`/`resumeToast`, both new exports) â€” per-toast timer state (`timerId`/`remaining`/`armedAt`) lives in a module-level `Map` in `toast.ts`, deliberately kept off the reactive `Toast` object so the template only ever sees `id`/`message`/`type`. No call site changes were needed anywhere â€” the improvements are entirely internal to `showToast`/`ToastHost.vue`.
- **The supervisor is tied to the company, never a manually-picked person** (this supersedes the earlier "Supervisor-by-company filtering" design, where the coordinator picked company AND supervisor as two independent dropdowns): `EnrollmentService::enrollOrReactivate()` no longer takes a `$supervisorId` parameter at all â€” every caller supplies only a `$companyId`, and the service derives `supervisor_id` itself from `Company::loginSupervisor()->user_id`, aborting 422 ("This company has no supervisor account yet...") if the company has none. ## Architecture Notes
- **Routing**: `routes/api.php` defines `/api/user` (auth:sanctum) plus an `admin/*` group gated by `auth:sanctum` + `role:admin`. Role gating middleware is `App\Http\Middleware\EnsureRole`, aliased as `role` in `bootstrap/app.php`, used as `->middleware('role:admin')` or `role:admin,coordinator` for multi-role routes. It also blocks `is_active = false` accounts with a 403 before checking role.
- **Controllers**: `app/Http/Controllers/Admin/*` for admin-only endpoints, `app/Http/Controllers/Auth/*` for Breeze/Sanctum auth flows. Follow the existing Form Request + Controller pattern (one Form Request per create/update action in `app/Http/Requests/{Admin,Auth}/`) before inventing new patterns.
- **Web SPA structure**: pages live in `web/src/pages/{admin,coordinator,student,supervisor}/`, role-specific shells in `web/src/layouts/*Layout.vue` wrapping the shared `web/src/components/layout/DashboardShell.vue`. Routes are defined centrally in `web/src/router/index.ts`, gated by `meta: { requiresAuth, role }` and a global `beforeEach` guard that calls the Pinia `auth` store (`web/src/stores/auth.ts`). API calls go through the shared Axios instance at `web/src/lib/axios.ts`.
- **Dev proxy**: `web/vite.config.js` proxies `/api`, `/sanctum`, `/login`, `/logout`, `/register`, `/forgot-password`, `/reset-password` to the Laravel backend (`VITE_BACKEND_URL`, default `http://localhost:8000`) so the Vite dev server at `:5173` can make same-origin requests.
- **CORS/Sanctum config**: `FRONTEND_URL` and `SANCTUM_STATEFUL_DOMAINS` in `.env` must match wherever `web/` is actually running â€” check `.env.example` when debugging auth/CORS issues.
- **Deployment (production hosting)**: `web/` (the Vue 3 SPA) deploys to **Vercel** (static build â€” a good fit, since it's just a Vite-built static site). The Laravel API deploys **separately, on a different host** (not yet decided) â€” Vercel does not natively support PHP/Laravel: no persistent process for the scheduled console commands in `routes/console.php` (weekly bundling every Saturday, the nightly 2am archive purge, missing-journal reminders), an ephemeral function filesystem, and no built-in persistent database â€” so only the SPA half of this monorepo goes there. Because the SPA and API will live on different domains once deployed (unlike local dev, where `web/vite.config.js`'s proxy makes them same-origin), `FRONTEND_URL`/`SANCTUM_STATEFUL_DOMAINS` must be set to the real Vercel production domain, and Sanctum's session cookie needs `SameSite=None; Secure` for the cross-site credentialed requests to keep working. The API host is expected to have an **ephemeral filesystem** (container-based, not a persistent VPS disk) â€” this is why avatar storage should move off the local `public` disk to an S3-compatible provider (Cloudflare R2 recommended: free tier that doesn't expire, no egress fees, drop-in compatible with the `s3` disk already defined in `config/filesystems.php`) before that API deploy happens; not yet implemented as of this note.
- **Gotcha: `date`-cast columns and plain equality under SQLite.** MySQL's `DATE` column type truncates any time component on write; SQLite's does not (SQLite has no real column-type enforcement), so a `date`-cast Eloquent attribute written as `'2026-07-06'` can come back out of SQLite storage as `'2026-07-06 00:00:00'`. A plain `where('week_start', $dateString)` â€” or `updateOrCreate()`'s match array, which is the same thing internally â€” silently fails to find the row it just created, and a second write inserts a duplicate instead of updating. Always use `whereDate('week_start', $dateString)` for lookups against a `date`-cast column (already the pattern in `WeeklyLogController::show()`/`index()`); this bit both `WeeklyBundlingService::bundleWeek()` and `WeeklyLogController::store()` in the same session before being fixed the same way â€” check any new code that reads back a `date`-cast column it just wrote.

- **Security-audit hardening pass**: three fixes from a full-codebase security review. (1) Route params named `date`/`weekStart` (`journal-entries/{date}`, `journal-entries/{date}/pdf`, `weekly-logs/{weekStart}`, `weekly-logs/{weekStart}/pdf`, `weekly-logs/{weekStart}/submit`, `journal-activities/{student}/{date}`) are now constrained via `Route::pattern('date', '\d{4}-\d{2}-\d{2}')`/`Route::pattern('weekStart', ...)` at the top of `routes/api.php` â€” previously a malformed segment reached the controller's unguarded `Carbon::parse()` and threw an uncaught exception (a stack-trace-leaking 500 if `APP_DEBUG` were ever left on), instead of 404ing at the router. (2) `AppServiceProvider::boot()` now throws a `RuntimeException` if `app()->environment('production')` and `config('app.debug')` are both true, since `.env.example` ships `APP_DEBUG=true` as a local-dev default with nothing else guarding against that value being carried into a real production `.env`. (3) `ProfileController::uploadPhoto()` re-encodes the stored avatar via GD (new private `reencodeImage()`, format-matched: `imagepng`/`imagewebp`/`imagejpeg`) immediately after `store()`, so a polyglot file (valid image container smuggling a non-image payload) can't survive on disk unmodified â€” defense in depth beyond the existing `image`+`mimes` validation in `UpdateProfilePhotoRequest`. No schema/route-shape changes; full test suite passes except the pre-existing weekend-flaky `SendMissingJournalEntryRemindersTest`.

- **Avatar upload pipeline overhaul (client-side crop + server-side standardization)**: `ProfileController::uploadPhoto()` no longer stores the uploaded file as-is with a light re-encode (the prior session's `reencodeImage()`, now removed as superseded); it now runs every avatar through a new `App\Services\AvatarProcessingService` (`sniffType()` + `toAvatarWebp()`) that (1) independently re-verifies the upload is a genuine JPEG/PNG/WebP by reading its actual image header via `getimagesizefromstring()` â€” a real magic-byte check, not the extension/declared-MIME-type trust that `UpdateProfilePhotoRequest`'s pre-existing `mimes:jpeg,jpg,png,webp|max:2048` rule alone provides (wired in via a new `withValidator()` hook on that FormRequest, so a spoofed-extension/MIME upload still 422s on `photo` even when the base `mimes` rule doesn't catch it â€” see `test_profile_photo_upload_rejects_garbage_bytes_disguised_as_an_image` in `ProfileControllerTest.php`), then (2) center-crops to a square, resizes to exactly 250Ã—250, and re-encodes as WebP (quality 82) via GD â€” re-encoding through a fresh `imagecreatetruecolor` canvas never carries EXIF forward, so metadata-stripping (including GPS) falls out of the resize step for free, no separate call needed. The stored filename is always a fresh `avatars/{Str::random(40)}.webp` â€” the original filename/extension is fully discarded, never reused. Upload order is write-new-then-delete-old (not the previous delete-then-write), so a mid-request failure can't leave a user with no avatar.
